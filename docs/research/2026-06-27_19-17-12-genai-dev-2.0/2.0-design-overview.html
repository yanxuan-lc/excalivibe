gen-ai-development · 2.0 · alpha

不是「跑哪些节点」，而是
这件事能放手到什么程度。

2.0 的核心是一个自治控制器（Autonomy Controller）：对每个任务先判断它能无人值守到什么档位，再据此组装轨道、人审门的形状、与验证强度。范式收敛为 SDD + TDD，落在「三层架构 + 专人专事 subagent + 独立功能 skill」之上。

可达的一半

可逆 / 意图明确的工作（视觉、确定性 bug、有覆盖的重构、通用 CRUD）→ 在闭环基建就位后趋向全自动。

不可约的一半

新颖核心 / 不可逆的工作 → 保留人对「运行中的产物」反应的回路，以及对不可逆对外动作的同意。这是类别，不是成本。

脊柱 · Tier-3 编排

自治控制器：三信号 → 四输出

任务起步时分类一次，把四件事写进 PIPELINE.md（可压缩续作的事实源）。它只编排，不实现任何能力。

① 变更原型

feature · bug · visual · refactor · schema-migration · research · docs · dependency-bump

→ 决定走哪条轨道（spine）

② 关键度 / 新颖度

core · supporting · generic

→ 深度档：设计思考与人工定型给多少

③ 可逆性 / 影响面

reversible · irreversible（共享 schema / 已发布契约 / 破坏性迁移 / 对外发布）

→ 自治档位（ceiling）

铁律：关键度只是深度档，永不切换轨道——核心域的 bug 仍是 bug，不升级成 feature。
单向升档：碰到不可逆面，无论原型一律就那一片升到人审。异常自动降档：异常率超注意力预算时，控制器把车道往人审方向拉回（降快升慢、带迟滞）。

四输出 = 节点图（轨道）· 自治档位 · 门禁形状 · 验证强度向量。绑定机制：每个产物门读 PIPELINE.md + 盘上产物——轨道靠门强制，不靠记忆。

自治档位

三档：放手多少，人审就有多少

full-auto 全自动

无人在环；仅限可逆动作 + 机器 oracle。须项目具备 canary/遥测/auto-revert 闭环才授予，否则降为抽检。

auto + spot-check 抽检

自动发布，预算内事后抽样复核；确认坏样本触发 auto-revert。抽样针对系统性漂移，非偶发灾难。

human-gated 人审

不可逆动作前人介入。核心工作走两道门 + 发布同意（见下）。

merge 到集成分支（可逆）见绿自动；只有 publish（不可逆对外）必须人同意——这去掉了「每条流程一道门」的假象。

人审 · 本轮重点

两道门，判不同的事，不可互替

核心工作的人审拆成两道——一道评结构（建得好不好），一道评行为（是不是要的东西）——加上对不可逆对外动作的发布同意。

架构门（Architecture Gate）

写代码前 · 评结构合理性与可扩展性 · 此刻改动只值一次 spec 编辑

载体 = 四层 REVIEW.md（下），由 spec-review skill 从 spec 单向派生，按深度档缩放。

意图门（Intent Gate）

对运行切片反应 · 评行为意图 · 抓「这不是我想要的」

默认载体 = 抛弃式原型，迭代到「对，就是这个」→ 再走正式管线建真品；原型绝不静默转生产。行为意图类决策（如「超额硬拦还是缓冲」）留这里定。

架构门载体：REVIEW.md 的四层

承载 markdown；内容形式优先级 图(UML) > 表格 > DSL(DDL) > 文字。自驱流程不假设你读过任何文档，名词讲清、术语首现「中文（English）」。

①
框定层 — 背景与范围 · 目标 / 非目标（Goals / Non-Goals）· 约束。给评审一把标尺。
②
结构层 — §0 领域模型（参考框架：限界上下文 + 聚合不变式 + 状态机），其后 §1 模块 · §2 接口 · §3 数据库 · §4 用例，逐一对照 §0 评。
③
审议层（ADR 式） — 关键决策的备选 + 理由 + 权衡。← 让「评得动合理性」的关键：只有看到「为什么这么选、否决了什么、代价是什么」，人才评得动。
④
横切层 — Tier A 失败一致性 · 安全/信任边界 · 性能/容量 · 可观测；Tier B 并发 · 兼容 · 安全性。每项 = 可测质量场景（当→则→度量）+ 权衡。
可逆 / generic 车道免两道重门：架构门靠深度档缩到很轻甚至无；没有意图门（你在已上线应用上的调整就是反馈）。

验证系统 · 信任的来源

验证三角 + 横切门 + 事后审计

验证三角（写权分离）

developer写产品码 + 单测；不写 e2e
qa-author从 spec 场景 派生 e2e；不碰产品码
e2e-runner双只读执行；验 UI 且 验库写
code-reviewer两判定（合规≠质量）；不信报告、从 diff 重推

干净上下文复核、文件路径交接——这才让「绿色报告」可信。

在流门 + oracle + 审计

在流门：security-scan · a11y-check · perf-budget（按面触发）。

信任信号：mutation / property oracle，不用行覆盖率。

抗模型单一：门禁车道 + 事后审计用异模型族。

事后审计：对已自动发布的变更无偏抽样，抓「自信但错」——遥测看不见、模型自评不可靠，只有它能抓。预算压力下绝不先砍它。

安全网 · 闭环 · 落地顺序

机械护栏与依赖顺序

PreToolUse 护栏 hook

控制器「不可逆动作必须有人」的确定性兜底。subagent + 不可逆对外 → deny；其余危险动作 → ask（交互问人 / headless 自动挡）；可逆操作放行。Claude 侧独有、非承载。

后置闭环

canary 分级放量 + 遥测看护 + auto-revert。这是让任何车道真能全自动的执行器——错的自动合并被抓回、撤销，无需人在环。

Phase A → B → C

A（现在）控制器 + 拆 merge/publish + 在流门 + oracle。B 建腿：闭环、多模型验证、注意力预算、意图门。C 按 B 就绪进度给可逆车道去门。

交付边界（诚实说）：这是个 markdown skills/agents/manifest 插件。后置闭环是「项目有 CD 才编排进去」，插件造不出。无 CD/遥测/回滚时，可逆车道停在抽检，不升全自动——全自动由项目的基建挣得，不是插件给的。

相对位置 · 诚实

对 superpowers 6.0 / matt 是「更全面」，不是「全面超过」

决定性领先：动态路由（自治档位组装轨道）、验证隔离（四上下文写权分离的三角）、优化侧流（smell-scan + 在流安全/无障碍/性能门）、完整调研子系统——两家都没有。

仍落后：成熟度（对手在跑、跨多 harness 实测；2.0 是 alpha，闭环/全自动多为纸面）、简洁度与上手摩擦、已验证的可移植性（Codex 端口未建）。

立场不同：matt 的极简是立场、superpowers 的克制是纪律；在它们自己的目标（精简·人在环·今天能跑）上没被打败。唯一实打实碾压的是验证三角——因为它从在跑的 1.x 继承来，不是画的饼。
