# REVIEW.md 模板与逐节写作指引

本文件是 planner 生成 `openspec/changes/<id>/REVIEW.md` 的权威模板。骨架照抄，
占位内容按指引填。成品参考 [worked-example.md](worked-example.md)。

写作总原则：

- **读者是要拍板的人，不是要实施的人。** 每节回答"这个设计对不对"所需的最少
  信息；执行级细节（边界值、字段校验规则、重试策略）留在 spec 里，节末给指针。
- **表格 > 图 > 散文。** 散文只用来点出"关键决策"——那种用户不确认、后面会吵
  起来的点。
- **缺席显式标注，编号恒定。** 某契约本变更不涉及时，概览表里标"本变更不涉及"，
  正文**保留该节及其编号**、节内只写一行「本变更不涉及」。不许顺延编号——§1–§4
  与概览表、页内锚点必须恒定对齐。
- **每个关键决策点名"代价"。** 用户确认的是 trade-off，不是结论。
- **篇幅护栏。** worked-example 的体量是**上界不是目标**：整篇控制在一屏滚动
  量级，每节"关键决策"散文不超过一段；写不下的内容说明它属于 spec，下沉回去、
  节末留指针。REVIEW.md 读不完，spec-confirm 就退化成签字仪式。

## 逐节指引

### 头部

- 「Spec 版本」必须是 `scripts/spec-hash.sh openspec/changes/<id>/` 的实际输出，
  不许手编——Gate 1 会用同一脚本重算比对。**先看退出码**：非零（spec 还没写、
  目录不对）就不要生成 REVIEW.md，先把 spec 补齐。
- 「Spec 版本」行的格式是强约束：**反引号包裹的恰好 12 位十六进制**——Gate 1
  的提取配方（`grep -oE '[0-9a-f]{12}'`）依赖它，别加前缀、注释或改格式。
- 「生成时间」用 `date '+%Y-%m-%d %H:%M'` 取，不要凭记忆写。

### §1 模块

- 目录树只画**增量**及其上下文，不要全仓库铺开；greenfield 项目画完整首版布局。
- 依赖图的价值在**禁止方向**：把"谁不许依赖谁"画出来，这是 code-review 时最常
  吵的点。

### §2 协议

- JSON 示例字段值要有真实感（金额、时间戳、ID），用户靠例子核对语义。
- 错误码表的「对应场景」列回链 §4 的 S-ID——这是协议与验收口径互相校验的地方。

### §3 库表

- DDL 必须是已按 `dba-guideline` 自查后的版本，并在引语里声明自查覆盖项
  （PK / NOT NULL+DEFAULT / 记账三列 / 金额 DECIMAL / 注释）。
- 「关键决策」点名唯一约束 / 索引选择背后的业务语义，以及将来改变它的迁移代价。

### §4 用例

- 表格里 WHEN/THEN 写**摘要**，执行级三段式留在 spec；两者靠 S-ID 对齐。
- 开头那句"确认本节 = 确认验收口径"必须保留——这是 spec-confirm 的法律效力所在。
- 时序图只画主路径（通常是 S1），异常路径靠表格行表达。
