# REVIEW.md 模板与逐节写作指引（架构门人审文档）

本文件是 planner 生成 `openspec/changes/<id>/REVIEW.md` 的权威模板。REVIEW.md 是
**架构门（Architecture Gate）**的人审载体：在写代码之前，由人评估这次变更的**架构边界
是否合理、是否可扩展、是否守得住领域规则**——此刻改动只值一次 spec 编辑，而非重写。

骨架照抄，占位内容按指引填。成品参考 [worked-example.md](worked-example.md)。

---

## 三条硬要求（贯穿全文，不可违背）

### 1. 承载与内容形式
- **承载**：纯 markdown 文件，用户在自己的 markdown 查看器（IDE 预览 / Git 平台）直接读。
  **不生成 HTML、不起 HTTP 服务、不写自定义页面**——REVIEW.md 本身就是审阅载体。
- **内容形式优先级（高→低）**：**图（UML / mermaid）＞ 表格 ＞ DSL（如建表 DDL）＞ 文字描述**。
  每个要表达的点，先问「能不能用图说清」，不能再退表格，再退 DSL，文字是最后手段。
  - 结构/依赖/状态/时序 → 图（mermaid `classDiagram` / `erDiagram` / `stateDiagram-v2` /
    `sequenceDiagram` / `flowchart`）。
  - 并列属性、对照、清单 → 表格。
  - 精确的 schema / 契约 → DSL（```sql、```jsonc）。
  - 只有「关键决策的理由与权衡」这类无法图表化的，才用文字，且控制在一段内。

### 2. 自包含——不假设读者看过任何其他文档
本流程是 **AI 自驱（AI-driven）**的：用户**没有**读过 spec、proposal、BRIEF 或任何中间产物。
所以 REVIEW.md 必须**独立成篇**：
- 不要写「见 spec 第 X 节」「如 proposal 所述」就当读者已知——确需深究的，在「深究指针」里
  一句话说明那是什么、在哪。
- 出现的每个领域名词、缩写、外部系统名，**首次出现处用一句话讲清它是什么**。宁可啰嗦。

### 3. 术语首现双语
任何技术/领域术语**首次出现**时写成**「中文（English）」**，方便对照与检索。例：
「限界上下文（Bounded Context）」「聚合（Aggregate）」「幂等（Idempotency）」
「最终一致（Eventual Consistency）」。同一术语后续可只用中文。

## 深度档（Depth Dial）——按变更缩放，别一律满档

REVIEW.md 的体量随变更的**关键度/新颖度**缩放，不是每次都写满：

| 档位 | 写什么 |
|------|--------|
| **novel-core（新颖核心）** | **满档**：①框定 + §0 领域模型 + §1–§4 四契约 + ③审议 + ④横切（含可测场景）|
| **supporting（支撑）** | 框定一两行 + 实际涉及的契约 + 关键决策 + 触发到的横切面 |
| **generic / 可逆** | 框定一行 + 仅动到的契约；§0 领域模型常可省（无新领域概念）；横切按需 |

缺席的契约/层在概览表显式标注「本变更不涉及」，**保留编号**、节内只写一行。

## 骨架（照此结构生成）

````markdown
# 架构评审 — <change-id>

> 本文档由 spec（机器可读的设计产物）**单向派生**，仅供人工审查。修改意见请反馈给
> planner 回流至 spec 后重新生成，**请勿直接编辑本文档**（直接编辑会产生第二份事实源）。

| 元信息 | 值 |
|--------|-----|
| 变更 ID | `<change-id>` |
| 生成时间 | <YYYY-MM-DD HH:mm>（`date` 命令取）|
| Spec 版本 | `<scripts/spec-hash.sh 的输出，12 位十六进制>` |
| 关键度 | core / supporting / generic ｜ 自治档位 <ceiling> |
| Arch-review（AI 设计预审）| ✅ 已闭环（N 项已消化进 spec）/ [-] 跳过：<原因> |
| 状态 | ⏳ 待人工评审（架构门）|

**本文档分四层**：① 框定（背景边界）→ ② 结构（领域模型 + 四契约，描述「设计是什么」）
→ ③ 审议（关键决策的备选与权衡，评「为什么这么选」）→ ④ 横切（质量视角，每项给可测场景）。

**概览**

| 层 / 契约 | 本变更是否涉及 | 章节 |
|---|---|---|
| ① 框定 | ✅ | 见下 |
| ②§0 领域模型 | ✅ / 本变更无新领域概念 | §0 |
| ②§1 模块设计 | ✅ / 不涉及 | §1 |
| ②§2 接口设计 | ✅ / 不涉及 | §2 |
| ②§3 数据库设计 | ✅ / 不涉及 | §3 |
| ②§4 用例设计 | ✅ | §4 |
| ③ 关键决策 | ✅ N 条 | §5 |
| ④ 横切质量 | ✅ 触发 N 项 | §6 |

## ① 框定层

**背景与范围**：<这次变更要解决什么问题、改动覆盖到哪里。一两句，让没有任何前置
知识的读者能进入状态。出现的产品名/系统名首次出现处一句话解释。>

**目标 / 非目标（Goals / Non-Goals）**

| ✅ 目标 | ❌ 非目标（本可做、显式排除）|
|---|---|
| <要达成的> | <刻意不做的——这是堵住范围蔓延的关键>|

**约束（Constraints）**：<技术 / 资源 / 合规上的硬约束，它们是后续一切取舍的前提。>

## ② 结构层

### §0 领域模型（Domain Model）——后续四契约对照它评

> 这是评审的**参考框架**：模块/接口/库表/用例是否「合理」，都相对这里的领域模型而言。
> 本节承接需求澄清阶段已框定的领域概念，**不在此重新发明**。

- **限界上下文（Bounded Context）**：<本变更属哪个上下文；与相邻上下文的边界在哪、
  为什么不能渗过去。> （限界上下文 = 同一个词在不同业务范围里含义不同，故划定的语义边界。）
- **聚合与不变式（Aggregate & Invariant）**：用表格列出核心聚合及其**永远成立的规则**，
  并标注每条规则的**强制强度**（强一致 / 最终一致）。（聚合 = 一组必须一起保持一致的对象；
  不变式 = 任何时刻都必须为真的业务规则。）
- **状态与生命周期**：有状态的领域对象用 `stateDiagram-v2` 画出合法状态与转移。
- **领域关系**：用 `erDiagram` 或 `classDiagram` 画核心概念之间的关系。

### §1 模块设计（Module Design）

- **目录/包增量**：用 ```text 树展示新增（🟢）与存量（标注「存量，不变」）。
- **依赖方向**：用 `flowchart` 画允许的调用方向，**禁止方向用虚线 + ❌**——
  「谁不许依赖谁」是 code review 时最常吵的点，必须画出来。
- **职责边界表**：每个模块一行（边界一句话 + 对外入口符号）。
- 对照 §0：模块拆分是否对齐限界上下文？核心能力是否面向抽象（有无抽象接口/扩展点）？

### §2 接口设计（Interface Design）

> 评两件事：**(a) 业务领域划分是否合理**（接口有没有切在领域概念的关节上）；
> **(b) 扩展性 / 第三方接入**（未来扩展、对外开放 OpenAPI 的合理性与复杂度）。

- **接口面按关注点切分**：用表格列出各接口面（受众 / 操作 / 信任级别 / 变更频率）——
  切分本身就是领域划分的判断对象。
- **关键契约形状**：请求/响应用 ```jsonc 给真实感的示例值（不抄完整正式定义，会过时）。
- **抽象边界**：指出哪些内部接口屏蔽了实现细节（换实现不破坏调用方）——面向抽象的落点。
- **版本与扩展**：版本策略、可加字段不破坏（additive）、预留的扩展点、对外 spec（OpenAPI）。
- **统一约定**：错误信封、分页、幂等等跨接口约定。

### §3 数据库设计（Database Design）

- **总览**：`erDiagram` 画实体、关系、关键列（主键 PK / 唯一键 UK / 外键 FK + 一句注释）。
- **完整 DDL**：```sql 块，按 `dba-guideline` 自查后的最终版，并声明已自查项。
- **索引与扩展性**：表格列出每表索引构成、是否预算内、关键决策（唯一约束承担的业务语义、
  未来扩展的迁移代价）。
- 对照 §0：表结构是否映射聚合？不变式由哪个约束强制（还是只在应用层）？

### §4 用例设计（Use Cases）

> 用户确认本节即确认**验收口径（Acceptance Criteria）**：下列场景全部通过 = 本变更验收完成。
> （验收口径 = 判定「做完了没」的可观察标准。）

- **执行载体声明**：scripted（脚本化测试）/ agent-driven（由 AI 驱动执行）。
- **场景表**：稳定 ID（S1/S2…）+ WHEN（用户动作）+ THEN（可观察断言）+ DB 影响 + 载体。
- **主路径时序**：用 `sequenceDiagram` 画主场景（用户→前端→服务→DB）；异常路径靠表格行表达。

## ③ 审议层（关键决策 —— 评「合理性」看这里）

> 描述层（②）只说「设计是什么」；要评「合理吗」，必须看到**备选方案、选定理由、代价**。
> 每条关键决策一块，用「决策记录（ADR, Architecture Decision Record）」格式。
> （决策记录 = 把一个架构选择连同它的来龙去脉、备选与后果写下来的短记录。）

**D<n> · <决策标题>** · 状态：proposed / accepted

| 备选方案 | 利 | 弊 |
|---|---|---|
| <选项 A> | … | … |
| **✅ <选定项>** | … | **代价：<明确的牺牲>** |

**选定理由**：<为什么在本变更的目标/约束下它最优，一段内>。
**未决问题**：<留待你拍板的点，没有则写「无」>。

> 若某决策属于**行为意图**（如「超额是硬拦还是留缓冲」）而非结构选择，**不在架构门定**，
> 标注「→ 留意图门（对运行切片确认）」并简述，不要在这里替用户拍板。

---

## ④ 横切层（质量视角 —— 贯穿②的各契约，不单列成章）

> 横切关注点会从「模块/接口/库表/用例」的纵向缝里漏掉，故单列。
> **纪律**：每个触发的横切面给一个**可测的质量场景**——
> 「**当**<刺激/条件>，**则**<架构响应>，**度量**<可判定的数值/标准>」——再点名它带来的**权衡**。
> 空泛的「要快/要安全」不合格。

- **Tier A（默认触发，要免则说明理由）**：失败与一致性（Failure & Consistency）·
  安全与信任边界（Security & Trust Boundary，设计期威胁建模，区别于代码级扫描）·
  性能与容量（Performance & Capacity）· 可观测性（Observability）。
- **Tier B（条件触发，有该面才写）**：并发（Concurrency，与失败一致性同簇）·
  兼容与互操作（Compatibility）· 安全性（Safety，物理/受规管领域）。
  （领域状态模型已升格到 §0，不在此重复。）

每项格式：
> **<横切面>**：场景——当 …，则 …，度量 …；权衡——<为换取它而牺牲了什么>。

## 逐节指引（补充要点）

### 头部
- 「Spec 版本」必须是 `scripts/spec-hash.sh openspec/changes/<id>/` 的实际输出，不许手编——
  human-confirm 门（架构门的新鲜度校验）会用同一脚本重算比对。**先看退出码**：非零（spec 没写、
  目录不对）就不要生成 REVIEW.md，先把 spec 补齐。
- 格式强约束：**反引号包裹的恰好 12 位十六进制**（提取配方 `grep -oE '[0-9a-f]{12}'` 依赖它）。
- 「生成时间」用 `date '+%Y-%m-%d %H:%M'` 取，不要凭记忆写。

### §0 领域模型
- **承接，不发明**：领域概念在需求澄清阶段（grill 深档）已框定并播种到项目术语表
  `CONTEXT.md`。此处精炼成设计侧（聚合/不变式/状态机/上下文边界），保持术语一致
  （命名一致性由 `glossary-conformance` skill 机检）。
- 不变式的**强制强度**是关键评审点：标「最终一致」的不变式意味着可短暂越界——它通常和
  §3 的存储选型、§6 的失败一致性取舍直接挂钩，让评审者据此判断取舍值不值。

### ③ 审议层
- 没有「备选 + 代价」就不算决策记录，评审者评不动——宁可少写决策，不可只写结论。
- 理由必须和后果成对：不只写「选了 X」，要写「选 X 因为…，代价是…」。

### ④ 横切层
- 安全分清**设计期**（威胁建模 / 信任边界 / 数据分级）与**代码级**（`security-scan` skill 的
  SAST 扫描）——架构门评前者，不评后者，二者不重复。
- 性能/容量别漏：它是真实架构评审里被提最多的质量面。
